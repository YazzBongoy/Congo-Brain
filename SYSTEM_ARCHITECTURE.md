# System Architecture – IAGov

## Overview
The IAGov platform follows a **micro‑services, cloud‑native** architecture with the following layers:

1. **Presentation Layer** – UI (React SPA) and mobile clients.
2. **API Gateway** – Kong/Envoy handling routing, authentication (JWT via Keycloak) and rate‑limiting.
3. **Domain Services** – Bounded contexts implemented as independent services (e.g., `budget-service`, `resource‑service`, `corruption‑monitor`).
4. **Economic Engine** – High‑performance computation service (Python/Numba) exposing optimisation APIs.
5. **AI Engine** – LLM‑based agents (Claude/Claude‑3.5) for forecasting, RAG and policy simulation.
6. **Data Platform** – PostgreSQL OLTP, Snowflake/Redshift DW, and an S3‑based data lake.
7. **Event Bus** – Apache Kafka for async communication, event sourcing and CQRS.
8. **Observability** – Prometheus + Grafana, Loki for logs, OpenTelemetry tracing.

## C4 Model (high‑level)
- **Context**: Ministries, public enterprises, citizens, auditors interact via the API gateway.
- **Container**: Separate containers for each service, the event bus, the DB, and the AI engine.
- **Component**: Within `budget-service` you have `BudgetController`, `BudgetDomain`, `BudgetRepository` (Postgres), `BudgetProjection` (Kafka consumer).
- **Code**: Each component follows Hexagonal architecture – adapters for external APIs (e.g., payment gateways).
