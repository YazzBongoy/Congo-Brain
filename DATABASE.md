# Database Design – IAGgov 

## Overview
The IAGov platform **must** store a massive amount of heterogeneous data: fiscal budgets, procurement contracts, resource extraction logs, economic indicators, AI model outputs, and audit trails.

## Physical Data Store
- **PostgreSQL 15** – Primary OLTP store for transactional data (budgets, contracts, user accounts).
- **TimescaleDB extension** – Time‑series storage for macro‑economic indicators and sensor data (e.g., energy consumption).
- **MinIO (S3 compatible)** – Object storage for raw documents, PDFs, satellite imagery.
- **Snowflake / Redshift** – Data‑warehouse for analytical workloads (BI dashboards).

## Indexes & Partitioning
- **Partition `budget`** by `year` (range) for fast annual queries.
- **GIN index** on JSONB columns storing flexible KPI metrics.
- **BRIN index** on `extraction.timestamp` for massive time‑series logs.

## Security & Auditing
- **Row‑level security (RLS)** on `budget` and `contract` tables – only the owning ministry can read/write.
- **Audit triggers** on all tables writing to `audit_log` (who, when, operation, before/after JSON).
- **Encryption at rest** – Transparent Data Encryption (TDE) for PostgreSQL and S3‑bucket policies.

## Migration Strategy
1. Initialise empty schema (`hermes db init`).
2. Run Alembic migrations (`alembic upgrade head`).
3. Load legacy CSVs via `COPY` into staging tables, transform, and insert into production tables.
